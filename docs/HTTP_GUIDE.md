# ウェブの仕組み（HTTP）詳細ガイド

このガイドでは、ブラウザとサーバーがどのように会話しているのか、図解を交えて詳しく解説します。

---

## 1. 全体図：データの通り道（バケツリレー）

ブラウザでボタンを押してから、画面に表示されるまでの「往復の旅」です。

```text
 [ あなたのパソコン ]          [ サーバー (裏側) ]          [ データベース ]
 +--------------+            +----------------+          +--------------+
 |  1. ブラウザ  |  リクエスト    |   2. Go言語    |   SQL     |  3. データの  |
 | (画面・ボタン) | -----------> |   プログラム    | --------> |    保管場所   |
 |              | <----------- |                | <-------- | (Postgres)  |
 +--------------+   レスポンス  +----------------+  保存完了  +--------------+
      ^                                |
      | 4. 画面を新しく描き直す           | 2.5 ビジネスルールをチェック
      +--------------------------------+     (NGワードはないか？など)
```

---

## 2. HTTPリクエスト（注文票）の中身

ブラウザがサーバーに送る「注文票」の正体です。

```text
 +---------------------------------------------------------+
 | [1] メソッドとURL:  POST  /posts  HTTP/1.1              | <-- 何をどうするか
 +---------------------------------------------------------+
 | [2] ヘッダー:       Host: localhost:8080                | <-- 送信先の住所
 |                     Content-Type: application/json      | <-- データの形式
 +---------------------------------------------------------+
 | [3] ボディ (中身):                                      |
 |      {                                                  | <-- 実際のデータ
 |        "title": "はじめての投稿",                       |
 |        "author": "さわこう"                             |
 |      }                                                  |
 +---------------------------------------------------------+
```

---

## 3. HTTPレスポンス（お返し）の中身

サーバーから戻ってくる「お返し」の中身です。

```text
 +---------------------------------------------------------+
 | [1] ステータス:     201 Created                         | <-- 成功・失敗の返事
 +---------------------------------------------------------+
 | [2] ヘッダー:       Content-Type: application/json      | <-- お返しの形式
 +---------------------------------------------------------+
 | [3] ボディ (中身):                                      |
 |      {                                                  | <-- 保存されたデータ
 |        "id": 1,                                         |
 |        "title": "はじめての投稿",                       |
 |        "created_at": "2026-02-27..."                    |
 |      }                                                  |
 +---------------------------------------------------------+
```

---

## 4. サーバー内部の 4 層構造（役割分担）

Go言語のプログラム（サーバー）の中では、データがさらに細かくリレーされます。

```text
   (ブラウザからのリクエスト)
           |
           v
 +-----------------------+
 |  ① Handler (窓口)     |  <-- 届いたデータを「Goの言葉」に翻訳して受付
 +-----------|-----------+
           |
           v
 +-----------|-----------+
 |  ② Service (計算)     |  <-- 「NGワードはないか？」などのルールを判定
 +-----------|-----------+
           |
           v
 +-----------|-----------+
 |  ③ Repository (倉庫番) |  <-- データベースへの「書き込み方」を指示
 +-----------|-----------+
           |
           v
   (データベースへ保存)
```

---

## 5. ステータスコード（結果の返事）の覚え方

サーバーからの「3桁の数字」は、信号機のような役割です。

```text
 [ 2xx (青) ]  成功！
    - 200 OK      : うまくいきました。
    - 201 Created : 新しく作れました。

 [ 4xx (黄) ]  お客さん（ブラウザ）側のミス。
    - 400 Bad Request : 入力したデータが変です。
    - 404 Not Found   : そのメニュー（URL）はありません。

 [ 5xx (赤) ]  お店（サーバー）側のミス。
    - 500 Internal Server Error : プログラムにバグがあって止まりました。
```

---

## まとめ：なぜこんなに分かれているの？

- **ブラウザ**: 「見た目」をきれいにすることに集中する。
- **サーバー**: 「ルール」を守らせることに集中する。
- **データベース**: 「データ」を安全に保存することに集中する。

この 3 つが **HTTP という共通のルール** で会話することで、一つのシステムとして動いています。
