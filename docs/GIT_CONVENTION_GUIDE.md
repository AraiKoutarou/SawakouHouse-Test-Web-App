# Git 運用ルール（コミット・ブランチ）ガイド

このプロジェクトでは、あとで履歴を振り返ったときに「いつ、何のために、何を変えたのか」を誰でも一目でわかるようにするため、特定のルール（規約）に従って Git を操作します。

---

## 1. コミットメッセージのルール (Conventional Commits)

コミットメッセージの先頭に、変更の内容を表す **「プレフィックス（接頭辞）」** を付けます。

### 基本の形
```text
プレフィックス: 変更内容の短い要約
```

### よく使うプレフィックス一覧
- **`feat`**: (feature) 新機能の追加
    - 例: `feat: 投稿の削除機能を追加`
- **`fix`**: バグの修正
    - 例: `fix: 投稿一覧が読み込めない問題を修正`
- **`docs`**: (documentation) ドキュメントのみの変更
    - 例: `docs: Git運用ガイドを作成`
- **`style`**: コードの意味に影響しない変更（改行、スペース、セミコロンなど）
    - 例: `style: コードのインデントを整理`
- **`refactor`**: リファクタリング（機能は変えずに、コードをきれいに書き直すこと）
    - 例: `refactor: 投稿作成の処理を整理`
- **`test`**: テストコードの追加や修正
    - 例: `test: ログイン機能のテストを追加`
- **`chore`**: (雑用) ビルド設定やライブラリの更新など、コード本体以外
    - 例: `chore: package.json の依存ライブラリを更新`

---

## 2. ブランチ名のルール

`main` ブランチ（本番用のきれいなコード）を汚さないために、作業内容ごとに「枝分かれ（ブランチ）」を作って作業します。

### 基本の形
```text
種類/作業内容のキーワード
```

### よく使う種類（カテゴリ）
- **`feature/`**: 新機能を作るとき
    - 例: `feature/add-delete-button`
- **`fix/`**: バグを直すとき
    - 例: `fix/error-message-layout`
- **`docs/`**: ドキュメントを整えるとき
    - 例: `docs/update-readme`
- **`refactor/`**: コードを整理するとき
    - 例: `refactor/clean-handler-logic`

---

## 3. なぜルールを決めるのか？（初心者のためのQ&A）

### Q. なぜ `feat` や `fix` を付けるの？
A. **「履歴の検索性」**を上げるためです。
数ヶ月後に「あのバグを直したのはいつだっけ？」と探すとき、`fix` で検索すればすぐに見つかります。また、自動的に「アップデート情報（変更履歴）」を作るツールも使いやすくなります。

### Q. なぜブランチを分けるの？
A. **「安全に試行錯誤するため」**です。
直接 `main` をいじると、途中でコードが動かなくなったときに「動いていた状態」に戻すのが大変です。ブランチを分けていれば、失敗してもそのブランチを捨てるだけで済み、`main` の安全は守られます。

### Q. 英語で書かないといけないの？
A. 世界中の開発者が理解できるように英語（feat, fixなど）をキーワードに使うのが業界標準ですが、要約（メッセージの中身）は日本語で書いてもOKです（このプロジェクトでは日本語推奨）。

---

## まとめ：開発の進め方の流れ
1.  新しいブランチを作る (`git checkout -b feature/my-new-task`)
2.  コードを書く
3.  ルールに従ってコミットする (`git commit -m "feat: ○○機能を追加"`)
4.  終わったら `main` ブランチに合体（マージ）させる
5.  履歴がきれいな状態で保存される！
